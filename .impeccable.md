## Design Context

### Users
General Vercel conference attendees at the Workflow SDK GA night — mixed technical levels, watching a live projector demo from the back of a room. They need to see everything at a glance, understand what's happening in real time, and leave thinking "I need to try this SDK."

### Brand Personality
**Bold, technical, theatrical.** This is a stage demo, not a product dashboard. Every visual element exists to make the Workflow SDK feel powerful and immediate. The polish IS the message — if the demo looks like a real Vercel product, the SDK earns credibility by association.

### Aesthetic Direction

**Theme:** Dark-only. Black background (`bg-black`), zinc-950 card surfaces, white/10 borders. No light mode.

**Typography:**
- **Sans:** Geist (Vercel's system font) — all UI text
- **Mono:** Geist Mono — data values, IDs, step names, elapsed timers, event log
- **Scale (projector-optimized):** Body minimum `text-lg`, labels `text-xl`, card headings `text-3xl`, section headlines `text-6xl`. Never use `text-xs`, `text-sm` for anything the audience needs to read.
- **Labels:** `text-sm font-semibold uppercase tracking-[0.2em] text-zinc-500` — Vercel's signature label style (exception to size rule: labels are small by design, read by the presenter not the audience)

**Color System (semantic, not decorative):**
| Role | Color | Usage |
|------|-------|-------|
| Success | `emerald-400/500` | Completed steps, happy-path indicators |
| Error | `red-400/500` | Failed steps, failure-armed states |
| Waiting | `amber-300/400/500` | Hooks paused, pending human input |
| Compensation | `fuchsia-300/400/500` | Saga rollback events, compensation cards |
| Running | `sky-300/400` | Active step animation, in-progress |
| Primary | `white` | Active buttons, success node fills, text |
| Neutral | `zinc-400/500/600/700` | Secondary text, borders, inactive states |

**Brand mark:** The ▲ triangle is Vercel's logo. Use it ONLY as:
- Logo in the header/nav
- Success-state icon fill inside timeline nodes
- Never as decorative scattered elements — the triangle is a mark, not a motif

**Spacing:** Generous. `p-10` for card padding, `gap-10` between sections, `py-6` for list items. The extra whitespace reads well on projectors.

**Cards:** `rounded-2xl border border-white/10 bg-zinc-950 p-10` — the universal container. Conditional accent borders (fuchsia for compensations, amber for waiting states, red for armed failures).

**Buttons:**
- Primary: `bg-white text-black` or `bg-black text-white`, `rounded-xl`, `font-semibold`
- Destructive/reject: `border-red-500/40 text-red-400`
- Pill/nav: `rounded-full border border-white/10 bg-white/5 px-5 py-2`

**Status indicators:** Colored dots (`h-3 w-3 rounded-full`) next to labels. Green=happy, red=armed, amber=building.

**Phone mockup:** White interior, 14px black border, `rounded-[56px]`, dynamic island notch. The phone is a presentation prop — it's 560×1120px, intentionally oversized for projection.

**Timeline nodes:** 96×96 circles (`rounded-full`), not triangles. State colors map to the semantic color system above. Connected by 2px lines.

**Event feed:** Geist Mono, `text-lg`, three-column layout (timestamp, kind badge, message). Kind badges use `kindColor()` mapping.

### Presentation Format

This is a **live projector presentation**, not a web app. Every screen must be designed for a 1920×1080 HD projector.

**One concept per screen.** Each "slide" maps to exactly one Workflow SDK concept or demo moment. The audience should never have to parse two ideas at once. If a screen requires explanation beyond one sentence, it's doing too much — split it.

**Full-viewport by default.** Each screen should fill the entire 1920×1080 viewport. No scrolling during the presentation — the presenter clicks/navigates between discrete screens, not within them. Design each view as a self-contained, non-scrollable frame.

**HD constraints:**
- Target `max-w-[1920px]` with content filling the space confidently
- Avoid layouts that require the audience to scan vertically — everything important should be visible without scrolling
- When showing the phone mockup + dashboard side-by-side, both must be fully visible in a single viewport
- Code snippets (if shown) should be ≤15 lines, `text-2xl`+ in Geist Mono, syntax highlighted against dark background

**Presentation flow:**
- Screens transition by navigation (route changes or state transitions), not by scrolling
- Each screen has a clear visual hierarchy: one headline, one primary element (demo, code, or diagram), minimal supporting text
- The presenter drives the story; the screen reinforces it — never the other way around

### Design Principles

1. **Projector-first.** Every element must be legible from 30 feet away. When in doubt, go bigger. Minimum `text-xl` for anything the audience reads; `text-4xl`+ for anything that tells the story.

2. **One concept, one screen.** Each view teaches exactly one Workflow SDK idea. The audience should never wonder "what am I looking at?" — the answer should be immediate and singular.

3. **Vercel-authentic.** This should look like it ships inside `vercel.com/dashboard`. Same card styles, same label conventions, same spacing rhythm. If it wouldn't pass review on the Vercel design team, it doesn't belong here.

4. **Color earns its place.** Background is black. Cards are zinc-950. Text is white/zinc. Color only appears to encode state — success, failure, waiting, compensation, running. No decorative gradients, no ambient color.

5. **The triangle is a brand, not a joke.** ▲ appears as a logo mark and as a success icon inside timeline nodes. That's it. No triangle rain, no triangle borders, no scattered triangles.

6. **The demo tells the story.** Every UI element exists to make a Workflow SDK concept visible and visceral — step boundaries, durable hooks, compensation stacks, real-time streaming. If a visual element doesn't teach something about the SDK, remove it.
